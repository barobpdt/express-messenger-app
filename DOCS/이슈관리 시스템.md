# Jira-like Task Tracking System: Database Design & Implementation Plan

이 문서는 팀별 메신저 연동, 캘린더/대시보드 보기, 산출물(기획/분석/설계/개발) 버전 통합 관리 및 피드백 기능을 포함하는 Jira 형태의 Task 추적 시스템 구축을 위한 데이터베이스 설계와 구현 계획입니다.

## 1. 데이터베이스 설계 (ERD 기준)

### 1.1 핵심 테이블 구성

**1) Users (사용자)**
*   `id` (PK) - INT, Auto Increment
*   `team_id` (FK) - INT
*   `username` - VARCHAR(50)
*   `email` - VARCHAR(100)
*   `avatar_url` - VARCHAR(255)
*   `created_at` - TIMESTAMP

**2) Teams (팀 및 메신저 연동 정보)**
*   `id` (PK) - INT, Auto Increment
*   `name` - VARCHAR(100)
*   `webhook_url` - VARCHAR(255) : 메신저 연동 웹훅 URL (슬랙, 잔디, 사내 메신저 등)
*   `channel_id` - VARCHAR(50) : 알람을 보낼 메신저 채널 ID
*   `created_at` - TIMESTAMP

**3) Projects (프로젝트)**
*   `id` (PK) - INT, Auto Increment
*   `name` - VARCHAR(100)
*   `description` - TEXT
*   `status` - ENUM ('active', 'archived', 'completed')
*   `created_at` - TIMESTAMP

**4) Tasks / Tickets (요청 티켓 / 업무)**
*   `id` (PK) - INT, Auto Increment
*   `project_id` (FK) - INT
*   `reporter_id` (FK) - INT : 요청자
*   `assignee_id` (FK) - INT : 담당자
*   `title` - VARCHAR(200)
*   `description` - TEXT
*   `status` - ENUM ('todo', 'in_progress', 'review', 'done', 'hold')
*   `priority` - ENUM ('low', 'medium', 'high', 'urgent')
*   `progress` - INT (0~100) : 진도율
*   `start_date` - DATE
*   `due_date` - DATE
*   `created_at` - TIMESTAMP
*   `updated_at` - TIMESTAMP

**5) Task Histories (업무 상태 및 변경 이력 - 메신저 알림용)**
*   `id` (PK) - INT, Auto Increment
*   `task_id` (FK) - INT
*   `user_id` (FK) - INT : 변경자
*   `field_changed` - VARCHAR(50) (e.g., 'status', 'assignee')
*   `old_value` - VARCHAR(255)
*   `new_value` - VARCHAR(255)
*   `changed_at` - TIMESTAMP

**6) Artifacts (산출물 관리)**
*   `id` (PK) - INT, Auto Increment
*   `task_id` (FK) - INT (또는 project_id)
*   `type` - ENUM ('planning', 'analysis', 'design', 'development') : 산출물 유형 (기획,분석,설계,개발)
*   `title` - VARCHAR(200)
*   `current_version` - INT : 현재 활성 버전 번호
*   `created_at` - TIMESTAMP

**7) Artifact Versions (산출물 버전 관리)**
*   `id` (PK) - INT, Auto Increment
*   `artifact_id` (FK) - INT
*   `version_number` - INT
*   `file_url` - VARCHAR(255) : 첨부파일 링크 또는 문서 경로
*   `content` - TEXT : 마크다운 등 본문 내용
*   `uploader_id` (FK) - INT
*   `uploaded_at` - TIMESTAMP

**8) Artifact Feedbacks (산출물 피드백/의견)**
*   `id` (PK) - INT, Auto Increment
*   `artifact_version_id` (FK) - INT : 특정 버전에 대한 피드백
*   `user_id` (FK) - INT
*   `content` - TEXT
*   `status` - ENUM ('open', 'resolved') : 피드백 반영 여부
*   `created_at` - TIMESTAMP

## 2. 구현 계획

### Step 1: 백엔드 및 DB 초기 구축 (Foundation)
*   **DB 스키마 생성**: 확장성을 위해 PostgreSQL 또는 MySQL 사용.
*   **메신저 연동 모듈 (Webhook/Bot)**:
    *   새 티켓(`Tasks`) 생성 시 `Teams.webhook_url`로 알림 전송 (수신자: 팀 채널).
    *   `Tasks.status` 변경(예: in_progress -> review) 시 담당자/요청자에게 자동 멘션 포함하여 메신저 알림 전송.

### Step 2: 캘린더 및 대시보드 API 생성
*   **캘린더 데이터 (Calendar API)**:
    *   `Tasks` 테이블의 `start_date`와 `due_date`를 조회. 팀별, 프로젝트별 필터링 기능 구현.
*   **대시보드 통계 (Dashboard API)**:
    *   **진도율**: (프로젝트 내 완료된 티켓 수 / 전체 티켓 수) * 100 또는 각 Task의 `progress` 평균.
    *   **상태별 분포**: Todo, In Progress, Done 개수 집계.
    *   **처리결과 요약**: 최근 주간 완료 티켓 및 지연 티켓 알림.

### Step 3: 산출물 버전 관리 및 피드백 기능 구현
*   **산출물 등록/버전업**: 새로운 문서(기획, 변경된 설계서 등) 업로드 시 `Artifacts`의 `current_version` 증가, `Artifact Versions`에 새 레코드 추가.
*   **버전별 피드백 관리**: `Artifact Feedbacks`를 통해 특정 버전 문서의 특정 위치나 전체 내용에 대한 코멘트 스레드 생성. 해결(Resolved) 시 상태 변경 기능 추가.

### Step 4: 프론트엔드 (UI/UX 구현)
1.  **Kanban / List View**: 태스크 상태별 드래그 앤 드롭 이동(수정 시 메신저 자동 알림).
2.  **Dashboard View (Chart.js / Recharts)**: 각 프로젝트/팀의 진도율 파이/바 차트, 번다운 차트 시각화.
3.  **Calendar View (FullCalendar)**: 업무 시작일, 종료일 기반 일정 표시 (담당자별 색상 구분).
4.  **Artifact Document Page**: 좌측엔 산출물 버전 리스트 선택, 우측엔 해당 버전의 파일/내용 표시 및 하단 피드백 스레드.

### Step 5: 통합 및 릴리스 (Integration & Launch)
*   메신저 연동 실서버 테스트 (사내망/외부인터넷망).
*   Role & Permission 설정: 권한이 있는 자만 산출물 버전을 올리거나 피드백 승인 처리.

---

이 설계 구조를 바탕으로, 필요하다면 백엔드(Node.js/Express, Spring Boot 등) 환경의 기본 스캐폴딩(Scaffolding) 코드 작성 또는 구체적인 프론트엔드 대시보드 페이지 구현을 바로 진행할 수 있습니다. 
원하시는 개발 스택(예: React + Node.js 등)이나 집중적으로 먼저 만들고 싶은 기능을 알려주시면 코드로 구체화해 드리겠습니다!
